1. Überblick

malletma.de ist das öffentliche Frontend einer zweiteiligen Architektur. Es liest Event-Daten aus einer Supabase-Datenbank (EU Frankfurt), die vom separaten Admin-Backend gefüllt wird.

Kernprinzip: Alle Event-Daten leben ausschließlich in Supabase. Der Frontend-Code enthält keine hardcodierten Event-Listen. Alles ist dynamisch und via Admin pflegbar.
Dual Supabase Client: Das Frontend nutzt den Anon Key (getSupabase()) für öffentliche Reads. Der Service Role Key (getSupabaseAdmin()) wird ausschließlich server-seitig für Strava-Token-Management verwendet.

5. Datenfluss — Events-Seite

1
Server Component lädt Events via Anon Key

app/events/page.tsx ruft getSupabase() auf, selektiert nur benötigte Spalten (EVENT_COLUMNS) und filtert server-seitig auf status IN ('published', 'active'). RLS-Policies stellen zusätzlich sicher, dass nur veröffentlichte Events sichtbar sind.

2
Filtern und Anreichern

Upcoming/Past trennen, EnrichedEvent[] bauen: snake_case → camelCase, Lat/Lon aus erster Route-Koordinate, Länderflaggen + -namen via countryFlag()/countryName(), Fallback-Werte.

3
Client-Component übernimmt

EventsFilterableContent empfängt EnrichedEvent[], verwaltet Filter-State in React, rendert Timeline + Übersichtskarte + Event-Cards.

4
Karten/Charts lazy geladen

Leaflet (CARTO Tiles, kein API-Key) und Recharts werden mit dynamic(..., {ssr: false}) geladen.

6. Filter-System

Das Filter-System auf der Events-Seite hat zwei Dimensionen:

DimensionWerteQuelle
Rad-TypRennrad · Gravel · MTBDB: bike_type
LandDynamisch aus Events (Flagge + Ländername)DB: country + Intl.DisplayNames

Auto-Hide-Regel

Eine Filter-Dimension wird ausgeblendet, wenn weniger als 2 Optionen einen Count > 0 haben. Gibt es z.B. nur Rennrad-Events, erscheint der Rad-Filter nicht.

Empty-Prevention

Klickt man auf die letzte aktive Pill einer Dimension, werden alle Optionen wieder aktiviert. Die Ergebnisliste bleibt nie leer.

Modus-Badge (Event-Cards)

type === 'race' → „Rennen“ (rot) · type === 'training' → „Training“ (amber) · type === 'charity' → „Charity“ (blau) · sonst → „Gran Fondo“ (primary)

8. Strava-Integration

1
Token aus DB lesen

getAccessToken(email) in lib/strava.ts liest access_token, expires_at, refresh_token aus oauth_tokens via Service Role Key.

2
Auto-Refresh wenn abgelaufen

Ist expires_at < now(), wird automatisch ein neuer Token bei Strava angefragt und in der DB aktualisiert (Upsert).

3
Aktivitäten abrufen

getStravaActivities(email) holt die letzten 30 Aktivitäten. Cache: revalidate: 300 (5 Minuten).

Owner-Email: STRAVA_OWNER_EMAIL aus Umgebungsvariablen bestimmt, wessen Strava-Daten angezeigt werden.

9. Sicherheit

🔑
Dual Supabase Keys

Frontend nutzt Anon Key (RLS-gefiltert) für öffentliche Reads. Service Role Key nur server-seitig für Strava-Tokens.

🗄️
Row Level Security (RLS)

Alle Tabellen haben RLS-Policies. Selbst wenn der Anon Key kompromittiert wird, sind nur veröffentlichte Events und Fahrräder lesbar.

🔒
Security Headers

HSTS (2 Jahre, includeSubDomains, preload), X-Content-Type-Options: nosniff, X-Frame-Options: DENY, X-XSS-Protection, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy (kein Kamera/Mikro/Geo). Konfiguriert in next.config.ts.

🔐
Secrets Management

Alle Secrets in Vercel Environment Variables. .env.local in .gitignore. Keine NEXT_PUBLIC_-prefixed Secrets — Service Role Key und Strava Client Secret nur server-seitig verfügbar.

📊
Spezifische Spaltenauswahl

Alle DB-Queries verwenden explizite Spaltenauswahl statt select('*'). Server-seitiger Status-Filter als zusätzliche Sicherheit neben RLS.

12. Glossar

BegriffErklärung
App RouterNext.js-System: Ordnerstruktur bestimmt URL-Routen. app/events/page.tsx → /events.
Server ComponentReact-Komponente auf dem Server gerendert. Hat DB-Zugriff, aber kein useState.
Client ComponentLäuft im Browser. Kann Interaktivität, braucht 'use client'-Direktive.
force-dynamicNext.js-Direktive: Seite wird bei jedem Request neu gerendert (kein Caching).
RLSRow Level Security — Supabase-Datenbankregeln. Events: nur published für anon. oauth_tokens: gesperrt.
Anon KeySupabase-Schlüssel mit eingeschränkten Rechten (RLS-gefiltert). Sicher für server-seitige Reads.
Service Role KeySupabase-Schlüssel der RLS umgeht. Nur server-seitig, nie im Browser.
JWTJSON Web Token — digital signierter „Ausweis“ für eingeloggte Nutzer.
LeafletOpen-Source JavaScript-Bibliothek für interaktive Karten. CARTO-Tiles, kein API-Key nötig.
RechartsReact-Diagramm-Bibliothek für Höhenprofile (Area-Chart).
EnrichedEventFrontend-Interface: DB-Zeilen (snake_case) in camelCase mit berechneten Feldern (Flagge, Koordinaten).
Intl.DisplayNamesBrowser/Node.js-API: „SE“ → „Schweden“ auf Deutsch.

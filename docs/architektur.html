1. Überblick — Was ist dieses System?

malletma.de ist eine persönliche Radsport-Website mit einem zugehörigen Admin-Backend. Es gibt zwei separate Next.js-Anwendungen, die dieselbe Supabase-Datenbank teilen:

🌐 Frontend — malletma.de

Öffentliche (aktuell per Auth0 geschützte) Website für die Cycling-Community. Zeigt den Race Calendar mit interaktiver Karte, Höhenprofilen, Countdowns und Filterbars. Außerdem: Strava-Live-Aktivitäten, Garage (Fahrräder) und Stats.

🛠️ Admin — malletma-admin.vercel.app

Internes Verwaltungs-Backend für Events. KI-gestütztes Erstellen und Aktualisieren von Racekalender-Einträgen: Agent scraped die Event-Website, extrahiert GPS-Route aus Komoot/RideWithGPS/GPX, berechnet Höhenmeter und generiert eine deutsche Beschreibung.

Kernprinzip: Alle Event-Daten (Name, Datum, Distanz, Route, Beschreibung, Land, …) leben ausschließlich in Supabase. Der Code enthält keine hardcodierten Event-Listen mehr. Alles ist dynamisch und via Admin pflegbar.

5. KI-Agent — Automatische Datenbeschaffung

Das Herzstück des Admins ist ein KI-Agent (admin/lib/agents/events.ts), der mit Claude Sonnet (Anthropic) betrieben wird und drei Tools hat:

🔍 scrape_webpage

Ruft eine URL ab, entfernt Scripts/Styles/Nav/Footer via Cheerio, gibt max. 8.000 Zeichen Textinhalt zurück. Wird für Event-Info-Seiten genutzt.

🗺️ extract_route_from_url

Extrahiert GPS-Route aus Komoot oder RideWithGPS URLs. Gibt distance_km, elevation_m, min/max_elevation_m und Punkt-Anzahl zurück.

📂 parse_gpx

Parsed GPX-XML direkt. Berechnet Distanz (Haversine), Höhenprofil (Ramer-Douglas-Peucker-Vereinfachung) und kumulativen Höhengewinn.

Auto-Difficulty

Nach dem Agent-Lauf (und nach Rescan) wird automatisch ein Schwierigkeitsgrad berechnet, wenn difficulty nicht vom Agent gesetzt wurde (admin/lib/difficulty.ts):

score = (distance_km / 40) + (elevation_m / 400) < 3.0 → easy (Leicht) 3–6 → medium (Mittel) 6–10 → hard (Schwer) > 10 → extreme (Extrem)

Beispiel: MSR 300 km / 2.800 Hm → score 14.5 → extreme.
Kann im Edit-Formular manuell überschrieben werden.

7. Datenflüsse — Wie laufen die Kernprozesse ab?

Event erstellen (Create-Flow)

1
Admin gibt URLs ein oder lädt GPX hoch

Event-Info-URL (z.B. vatternrundan.se), Routen-URL (z.B. ridewithgps.com/routes/…) und/oder GPX-Datei.

2
Agent-API-Route entscheidet

Ist ANTHROPIC_API_KEY konfiguriert? → Ja: Claude-Agent mit bis zu 8 Turns. → Nein: Direktes GPX-Parsing oder Route-Extraktion ohne KI.

3
Claude scrapet, extrahiert, generiert

Agent ruft scrape_webpage für Event-Info auf, extract_route_from_url für GPS, gibt JSON mit allen Feldern inkl. notes zurück.

4
Auto-Difficulty-Berechnung

Falls Agent keine difficulty gesetzt hat: computeDifficulty(distance_km, elevation_m) aus admin/lib/difficulty.ts.

5
Preview im Browser

Admin sieht alle extrahierten Felder, kann sie editieren, dann auf "In Datenbank speichern" klicken.

6
POST /api/events speichert in Supabase

Event ist in DB — aber noch nicht sichtbar (status: 'draft'). Mit "Publish" wird status: 'published' gesetzt und erscheint auf malletma.de.

Frontend-Rendering (Events-Seite)

1
Server Component lädt Events via Anon Key

app/events/page.tsx (Server Component, force-dynamic) ruft getSupabase() (Anon Key) auf, selektiert nur benötigte Spalten und filtert server-seitig auf status IN ('published', 'active').

2
Filtern und Anreichern

Upcoming/Past trennen, EnrichedEvent[] bauen (camelCase, Lat/Lon aus erster Route-Koordinate, Fallbacks).

3
Client-Component übernimmt

EventsFilterableContent empfängt EnrichedEvent[], verwaltet Filter-State in React, rendert Timeline + Karte + Cards.

4
Karten/Charts lazy geladen

Leaflet und Recharts werden mit dynamic(..., {ssr: false}) geladen — verhindert Server-Rendering-Fehler von Browser-APIs.

8. Filter-System — Dynamische Pill-Filter

Das Filter-System auf der Events-Seite hat zwei Dimensionen:

DimensionWerteQuelle
Rad-TypRennrad · Gravel · MTBDB: bike_type
LandDynamisch aus Events (Flagge + Ländername)DB: country + Intl.DisplayNames

Auto-Hide-Regel

Eine Filter-Dimension wird ausgeblendet, wenn weniger als 2 Optionen einen Count > 0 haben. Gibt es z.B. nur Rennrad-Events, erscheint der Rad-Filter nicht — er wäre nutzlos. Sobald mindestens 2 Typen vorkommen, erscheint der Filter.

Empty-Prevention

Klickt man auf die letzte aktive Pill einer Dimension, werden alle Optionen der Dimension wieder aktiviert. Die Ergebnisliste bleibt nie leer.

Country-Filter

Ländernamen werden dynamisch aus den ISO-Codes der Events generiert: countryName("SE") → "Schweden" (via Intl.DisplayNames(['de'])). Flaggen via Unicode-Regionalindikatoren: countryFlag("SE") → "🇸🇪" (kein Emoji-Lookup-Table nötig).

9. GPX & Routendaten — Verarbeitungspipeline

1
Eingabe: GPX-Datei oder URL

GPX-Upload über Admin-Formular (Multipart), oder Komoot/RideWithGPS-URL.

2
Parsing (admin/lib/parsers/gpx.ts oder route-extractor.ts)

GPX: Cheerio parsed XML, extrahiert <trkpt>-Koordinaten und Elevationen.
Komoot: Öffentliche API (/api/v2/tours/{id}).
RideWithGPS: JSON-Endpunkt, bevorzugt elevation_gain aus Metadaten.

3
Höhenprofil (admin/lib/parsers/utils.ts)

buildElevationProfile(): Berechnet Distanz (Haversine zwischen Punkte), vereinfacht Profil auf max. 200 Punkte (Ramer-Douglas-Peucker, ε=0.02 km), berechnet kumulativen Höhengewinn (Summe aller positiven Deltas ≥ 1 m).

4
Polyline-Vereinfachung

GPS-Punkte werden auf max. 300 Punkte reduziert (Ramer-Douglas-Peucker, ε=0.001°). Gespeichert als [[lat, lon], …] in Supabase JSONB-Feld.

5
Ergebnis: ParsedRoute

{ polyline, elevation_profile, distance_km, elevation_m, min_elevation_m, max_elevation_m } — wird in DB gespeichert.

10. Login & Sicherheit

Architektur-Sicherheit

🔐
Authentifizierung (Auth0)

Admin-Bereich ist durch Auth0 OAuth2 mit PKCE-kompatiblem Flow geschützt. Session-Token werden als JWT mit HS256 signiert und als httpOnly Cookie gespeichert (8h Ablaufzeit).

🛡️
Proxy Auth-Guard (Next.js 16)

proxy.ts validiert JWT-Session auf jeder Route. API-Routes geben 401 zurück, Seiten leiten zum Login weiter. Next.js 16 verwendet die proxy-Konvention statt der veralteten middleware-Konvention.

🛡️
Defense-in-Depth

Zusätzlich zum Proxy validiert jede API-Route die Session eigenständig via requireSession(). Bei Proxy-Bypass oder Fehlkonfiguration greifen die Route-Guards.

🛡️
OAuth CSRF-Schutz

Auth0 Login-Flow verwendet einen state-Parameter (UUID), der in einem httpOnly Cookie gespeichert und im Callback validiert wird.

Datenbank-Sicherheit

🗄️
Row Level Security (RLS)

RLS-Policies aktiv: events — nur published/active für anon-Key. oauth_tokens, users, agent_scan_log — kein anon-Zugriff. bikes — öffentlich lesbar. Service-Role-Key umgeht RLS für Admin-Operationen.

🔑
Dual Supabase Client Pattern

Frontend (malletma-web) nutzt den Supabase Anon Key für öffentliche Reads via getSupabase(). Service Role Key wird nur im Admin-Backend via getSupabaseAdmin() für Schreiboperationen und Strava-Token-Management server-seitig verwendet.

📊
DB-Performance-Indizes

Indizes auf events(status), events(date) und events(status, date) für schnelle Event-Abfragen. Frontend-Queries verwenden spezifische Spaltenauswahl statt select('*').

Input-Validierung

✅
UUID-Validierung

Alle API-Routes validieren UUID-Parameter mit Regex via isValidUUID() bevor sie an Supabase weitergegeben werden.

✅
Field Whitelisting

PUT/UPDATE-Operationen verwenden Whitelists (EDITABLE_FIELDS) — nur definierte Felder werden akzeptiert. Rescan-Apply Route filtert auf erlaubte Felder.

✅
Numerische Feldvalidierung

Numerische Felder (distance_km, elevation_m, etc.) werden server-seitig validiert und konvertiert.

Transport-Sicherheit

🔒
Security Headers

HSTS (2 Jahre, includeSubDomains, preload), X-Content-Type-Options: nosniff, X-Frame-Options: DENY, X-XSS-Protection, Referrer-Policy: strict-origin-when-cross-origin, Permissions-Policy (kein Kamera/Mikro/Geo). Konfiguriert in next.config.ts beider Apps.

🍪
Cookie-Sicherheit

Session-Cookie: httpOnly, secure (production), sameSite=lax, 8h Ablaufzeit. Kein JavaScript-Zugriff auf Session-Daten möglich.

Secrets Management

🔐
Environment Variables

Alle Secrets (Auth0, Supabase, Strava, Anthropic) sind in Vercel Environment Variables gespeichert. .env.local ist in .gitignore ausgeschlossen. Keine NEXT_PUBLIC_-prefixed Secrets — Service Role Key, Client Secrets und API Keys sind nur server-seitig verfügbar.

API-Sicherheit

🤖
Agent-Begrenzungen

Scraping auf 8.000 Zeichen pro Webseite begrenzt. Timeout von 8 Sekunden für externe Requests. User-Agent identifiziert den Bot. Claude Agent hat ein Maximum von 8 Turns gegen endlose Loops.

14. Glossar

BegriffErklärung
App RouterNeueres Next.js-System: Ordnerstruktur bestimmt URL-Routen. app/events/page.tsx → /events.
Server ComponentReact-Komponente, die auf dem Server gerendert wird. Hat Zugriff auf DB, aber kein useState.
Client ComponentLäuft im Browser des Nutzers. Kann Interaktivität (onClick, State), braucht aber 'use client'-Direktive.
force-dynamicNext.js-Direktive: Seite wird bei jedem Request neu gerendert (kein statisches Caching). Nötig für Live-Daten.
JSONBPostgreSQL-Datentyp für JSON-Daten mit Indexierung. Speichert z.B. die GPS-Koordinaten-Arrays.
JWTJSON Web Token — digital signierter "Ausweis" für eingeloggte Nutzer. Kann nicht gefälscht werden.
HttpOnly CookieBrowser-Cookie, auf das JavaScript keinen Zugriff hat. Schutz vor XSS-Angriffen.
SSR (Server-Side Rendering)HTML wird auf dem Server gebaut, nicht im Browser. Schnelleres erstes Laden, gut für SEO.
Haversine-FormelMathematische Formel zur Berechnung der Entfernung zwischen zwei GPS-Koordinaten auf der Erdkugel.
Ramer-Douglas-PeuckerAlgorithmus zur Vereinfachung von Kurven/Polygonen. Reduziert GPS-Punkte bei minimalem Informationsverlust.
